# v1.0 Stable Verification

Verification date: 2026-07-09

Evidence branches:

- `codex/v1.0-stable-verification`
- `codex/v1.0-final-operational-evidence`

Release under verification: `1.0.0`

Status: **endpoint smoke passed; operational UI auth blocker found**

The local source tree passes the full automated quality gate, and the Kelvin VM
now serves the v1.0 application build. Production/LAN auth is enabled and the
runtime status endpoint reports `ready`. The final operational verification
pass found that production auth protects the data APIs, but the current static
UI does not provide an authenticated bearer-token/session path for those API
calls. Final completion is blocked until the UI can run the operational
checklist with auth enabled.

## 3. Checks Not Accepted Yet

### Operational UI auth blocker

Observed on 2026-07-09 against:

```text
http://192.168.10.13:8000
```

The following UI pages return static HTML successfully and do not show server
errors:

| Page | Observed result |
|---|---|
| `/ui` | Page loads with chat shell. |
| `/ui/runs` | Page loads, but run data shows `Hiba a futások betöltésekor: Sikertelen betöltés`. |
| `/ui/approvals` | Page loads, but approval data shows `Hiba: Sikertelen betöltés`. |
| `/ui/audit` | Page loads, but audit data shows `Hiba a betöltéskor: Sikertelen API lekérdezés`. |
| `/ui/settings` | Page loads, but settings data shows `Hiba a beállítások betöltésekor: Sikertelen betöltés`. |
| `/ui/n8n` | Page loads, but n8n status shows `Hiba részletei: Sikertelen API lekérdezés`. |

The matching protected API calls without a bearer token return `401`:

| Method | Endpoint | Result |
|---|---|---|
| `GET` | `/api/v1/agent/runs` | `401` |
| `GET` | `/api/v1/security/audit` | `401` |
| `GET` | `/api/v1/settings` | `401` |
| `GET` | `/api/v1/n8n/health` | `401` |
| `POST` | `/api/v1/settings/test-email` | `401` |
| `POST` | `/api/v1/settings/send-summary` | `401` |

The frontend scripts call these protected routes with plain `fetch(...)`
requests and do not attach an `Authorization: Bearer ...` header. This is
consistent with the historical ADR note that the frontend did not yet receive
an API token. With `KELVIN_API_AUTH_MODE=required`, this prevents final
operator verification of runs, approvals, audit, settings, n8n health, test
email, and daily summary from the UI.

### Remaining rejected checks

- approval flow

- email test
- approval notification

- daily summary email
- n8n outage behavior

## 4. Required Remediation

Before marking v1.0 complete:

1. Add an operator-safe way for the local UI to authenticate protected API
   calls when `KELVIN_API_AUTH_MODE=required`, without storing raw tokens in
   Git or exposing them in page output.
2. Rerun the full endpoint, UI, notification, n8n outage, backup, and restore
   evidence checklist from `docs/operational-runbooks.md`.
3. Update this document with the successful live verification evidence.
4. Only then mark Step 9 and v1.0 complete in `docs/ai/v10-guide.md` and
   `docs/roadmap.md`.
